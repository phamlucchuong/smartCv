# [Feature] Implement Profile Manager Real API Hooks

## Overview

The Profile page (`_account.profile.tsx`) and Settings page (`_account.settings.tsx`) in `web-candidate` have fully built UIs, but every mutation is stubbed with `toast.info('... coming soon')`. All required API hooks are already generated by Orval. This issue covers wiring all mutations to live endpoints and adding avatar upload (requires a new backend endpoint).

## Reproduction Steps

1. Sign in as a candidate.
2. Navigate to `/profile`.
3. Click **Edit**, change any field (name, bio, address, etc.), click **Save** → `"Profile update coming soon"` toast.
4. Add/edit/delete a work experience, education, or certification entry → same stub toast.
5. Navigate to `/settings`.
6. Toggle any notification or privacy switch → `"Settings update coming soon"` toast.
7. Fill in Change Password form → `"Password update coming soon"` toast.
8. Fill in Email Address field → `"Email update coming soon"` toast.
9. Click **Delete Account** → signs out locally but does **not** call `DELETE /api/candidates/me`.

## Expected Behavior

All save, add, delete, and toggle actions persist to the backend immediately. Optimistic UI updates replace stub toasts. Avatar upload allows candidates to set a profile photo.

## Current Behavior

`useGetMe2()` and `useGetSettings()` fetch data on load (read-only works). Every write action fires a placeholder toast.

## Implementation Plan

### Pre-implementation: Understand the data model split

The backend separates profile data across two entities:
- **User entity** — `fullName`, `email`, `phone`, `password` (managed via `useUpdateUser` / `useChangePassword`)
- **Candidate entity** — everything else: `address`, `title`, `bio`, `experiences`, `educations`, `certifications`, etc. (managed via `useUpdate1`)

The profile page `draft` state (line 215 of `_account.profile.tsx`) is:
```ts
{ name: string; email: string; phone: string; location: string; title: string; bio: string }
```
Note: `draft.name` = full name, `draft.location` = address. These names differ from their API counterparts.

### 1. Profile Page — Basic Info Save

**File**: `frontend/apps/web-candidate/src/routes/_account.profile.tsx`

Because `fullName` and `phone` belong to the User entity, saving basic info requires **two concurrent mutations**:

**Hook A**: `useUpdate1` — `PUT /api/candidates/{id}` (candidate fields)  
**Hook B**: `useUpdateUser` — `PUT /api/users/{userId}` (user fields)

```ts
// candidateId from useGetMe2()?.data?.data?.id
// userId from useAuthStore().userId

const updateCandidateMutation = useUpdate1()
const updateUserMutation = useUpdateUser()

const handleSave = () => {
  const candidatePromise = updateCandidateMutation.mutateAsync({
    id: candidateId,
    data: buildCandidateRequest(draft, profile),
  })
  const userPromise = updateUserMutation.mutateAsync({
    userId,
    data: { fullName: draft.name, phone: draft.phone },
  })
  Promise.all([candidatePromise, userPromise])
    .then(() => {
      queryClient.invalidateQueries({ queryKey: getGetMe2QueryKey() })
      queryClient.invalidateQueries({ queryKey: getGetMeQueryKey() })
      setEditMode(false)
      toast.success('Profile saved')
    })
    .catch(() => toast.error('Failed to save profile'))
}
```

Replace the Save button's `onClick` stub (line 277) with `handleSave`.

### 5. Profile Page — Avatar Upload

#### Backend — Two new pieces required

**a. `S3Service.java` — new `uploadAvatar()` method**

`S3Service.uploadCv()` has a hard-coded PDF MIME type validator (`ALLOWED_CONTENT_TYPE = "application/pdf"`) that rejects images. A separate method is needed:

```java
// backend/user-service/src/main/java/.../config/S3Service.java
public String uploadAvatar(String userId, MultipartFile file) {
    validateAvatarFile(file); // accept image/jpeg, image/png, image/webp, max 2 MB
    String ext = extractExtension(file.getOriginalFilename());
    String key = "avatars/" + userId + "/" + UUID.randomUUID() + "." + ext;
    // Upload to S3 (same bucket as CVs — acceptable, key prefix differentiates)
    return "https://" + bucketName + ".s3." + region + ".amazonaws.com/" + key;
    // Return a permanent public URL (not presigned), OR store the raw key and presign on read.
}
```

> **Design decision — permanent vs presigned URL**: `S3Service.generatePresignedUrl()` produces time-limited URLs (default TTL: 60 min configured by `AWS_S3_PRESIGNED_URL_TTL_MINUTES`). If a presigned URL is stored as `avatarUrl` in MongoDB it will expire and the avatar will break. Two options:
> - **Option A (recommended)**: Make the `avatars/` prefix publicly readable in the S3 bucket policy and return a permanent URL.
> - **Option B**: Store only the S3 key (e.g. `avatars/userId/uuid.png`) and presign it on every `GET /api/candidates/me` response. Requires changes to `CandidateMapper`.
>
> Decide before implementing. The issue assumes Option A.

**b. `CandidateController.java` — new endpoint**

```java
@PostMapping("/me/avatar")
@PreAuthorize("hasRole('CANDIDATE')")
public ApiResponse<String> uploadAvatar(
    @AuthenticationPrincipal String userId,
    @RequestParam("file") MultipartFile file) {
    String avatarUrl = candidateService.uploadAvatar(userId, file);
    return ApiResponse.<String>builder().data(avatarUrl).message("Avatar updated").build();
}
```

Add `@Operation(summary = "Upload candidate avatar")` so springdoc generates a correct multipart schema and Orval produces a typed `useUploadAvatar` hook with `FormData`.

**c. `CandidateService.java` — delegate to S3Service**

```java
public String uploadAvatar(String userId, MultipartFile file) {
    String avatarUrl = s3Service.uploadAvatar(userId, file);
    Candidate candidate = getByUserId(userId);
    candidate.setAvatarUrl(avatarUrl);
    candidateRepository.save(candidate);
    return avatarUrl;
}
```

### 6. Settings — Notification Toggles

**Hook**: `useUpdateNotifications` — `PUT /api/candidates/settings/notifications`

The generated mutation signature is `mutate({ data: NotificationPreferences })`.

```ts
const updateNotifMutation = useUpdateNotifications()

const handleNotifToggle = (key: keyof typeof notifications) => {
  const next = { ...notifications, [key]: !notifications[key] }
  // Optimistic: update local state immediately
  setNotifications(next)  // requires lifting notifications to useState
  updateNotifMutation.mutate({
    data: {
      emailJobSuggestions:     next.jobRecommendations,
      emailApplicationUpdates: next.applicationUpdates,
      pushNotifications:       next.newMessages,
      marketingEmails:         next.promotionalEmails,
    },
  }, {
    onError: () => {
      setNotifications(notifications) // revert
      toast.error('Failed to update notifications')
    },
    onSuccess: () => queryClient.invalidateQueries({ queryKey: getGetSettingsQueryKey() }),
  })
}
```

> Currently `notifications` is derived directly from `settingsPayload` (no local state). Lift it to `useState` with the settingsPayload as initializer to enable optimistic toggling.

Replace `onToggle` props on lines 128–131 with `() => handleNotifToggle('jobRecommendations')` etc.

## Data Mapping Reference

### `buildCandidateRequest` helper

```ts
function buildCandidateRequest(
  profile: UserModels.CandidateResponse,
): UserModels.CandidateRequest {
  return {
    // Candidate-owned fields only (fullName/phone are NOT in CandidateRequest)
    address:     profile.address,
    title:       profile.title,
    bio:         profile.bio,
    avatarUrl:   profile.avatarUrl,
    skills:      profile.skills ?? [],
    yearsOfExperience: profile.yearsOfExperience,
    experiences: profile.experiences ?? [],
    educations:  profile.educations ?? [],
    certifications: profile.certifications ?? [],
    languages:   profile.languages ?? [],
    jobType:             profile.jobType,
    preferredLocation:   profile.preferredLocation,
    expectedSalaryMin:   profile.expectedSalaryMin,
    expectedSalaryMax:   profile.expectedSalaryMax,
    portfolioUrl:  profile.portfolioUrl,
    githubUrl:     profile.githubUrl,
    linkedinUrl:   profile.linkedinUrl,
  }
}
```

When saving basic info, override only the changed fields:
```ts
data: {
```

## Notes

- **`fullName` / `phone` split**: `CandidateRequest` does not include these fields. Saving name/phone requires calling both `useUpdate1` AND `useUpdateUser` concurrently. The profile page Basic Info save handler must fire both mutations.
- **`Education` model**: has only `startYear: number` and `endYear: number`. The `EduForm.startMonth` / `endMonth` strings captured in the UI have no backend equivalent — discard them in the mapping.
- **`WorkExperience.achievements`**: the `ExpForm` captures `achievements: string[]` but the `WorkExperience` model has no `achievements` field. Map the content to the `description` field (join as newline-separated string).
- **Avatar URL permanence**: must resolve Option A (public S3 prefix) vs Option B (presign on read) before implementation. Storing presigned URLs in MongoDB will cause avatar images to expire after ~60 min.
- **Avatar springdoc annotation**: add `@Operation(summary = "Upload candidate avatar")` and `@Parameter` annotations to the new endpoint so springdoc generates a correct `multipart/form-data` schema. Without it, Orval may generate `useUploadAvatar` with an `unknown` body type.
- After the avatar endpoint is added and deployed, run `pnpm gen:api` to regenerate the `useUploadAvatar` hook before implementing the frontend avatar section.
