# CV Scoring Flow Based on Existing JD

## Goal

Build a CV scoring pipeline that is explainable, deterministic in its final score,
and only uses LLMs for extraction/normalization tasks instead of letting the LLM
invent the final match score.

The target outcome is:

1. A candidate uploads a CV file (`pdf`, `docx`, or text).
2. The system extracts structured CV data into a fixed skeleton.
3. The system extracts structured requirement data from the job description.
4. The system computes the final score with a transparent scoring formula.
5. The API returns both:
   - the final score
   - the evidence and breakdown that explains why the score is what it is

This design avoids the current black-box approach where the LLM directly returns
`matchScore` without a verifiable calculation path.

---

## Problems With The Current Flow

Current behavior in `ai_engine_service`:

1. CV text is extracted.
2. JD data is fetched.
3. The prompt asks the LLM to compare CV and JD.
4. The LLM directly returns `matchScore`, `matchedSkills`, `missingSkills`, and summary.

Limitations:

- The score is generated by the model, not by the platform.
- The same CV/JD pair can receive slightly different scores across calls.
- It is difficult to prove why the score is 62 instead of 74.
- There is no strict audit trail for each scoring factor.
- Business teams cannot easily tune the weighting logic.

The LLM is useful for extraction and normalization, but it should not be the final
source of truth for the score.

## Recommended Target Flow

### Phase 1. CV ingestion

Input:

- uploaded CV file (`pdf`, `docx`) or CV text

Processing:

1. Extract raw text from the CV.
2. Normalize formatting noise.
3. Preserve section boundaries when possible:
   - summary
   - experience
   - skills
   - education
   - certifications
   - projects

Output:

- `rawCvText`
- optional `documentMetadata`

Notes:

- For `docx`, prefer deterministic text extraction before any LLM step.
- Keep the raw extracted text for audit/debugging.

### Phase 2. CV structured extraction with LLM

Use LLM for one task only:

- map free-text CV into a strict skeleton

Example CV skeleton:

```json
{
  "candidateProfile": {
    "targetRoles": [],
    "seniorityLevel": "",
    "domains": [],
    "yearsOfExperience": 0
  },
  "skills": {
    "technical": [],
    "tools": [],
    "frameworks": [],
    "databases": [],
    "cloud": [],
    "softSkills": [],
    "languages": []
  },
  "experience": [
    {
      "title": "",
      "company": "",
      "durationMonths": 0,
      "responsibilities": [],
      "achievements": [],
      "evidenceSkills": []
    }
  ],
  "education": [],
  "certifications": [],
  "projects": []
}
```

Requirements for this step:

- force JSON output only
- validate against a schema
- retry on schema failure
- keep both:
  - extracted structured CV
  - extraction confidence or validation flags

Best practice:

- ask the LLM to extract facts only, not opinions
- forbid score generation in this step
- require every important field to be grounded in CV text

### Phase 3. JD requirement extraction

Do not score directly from raw JD text.

Instead, create a structured JD requirement object first.

Example JD skeleton:

```json
{
  "jobInfo": {
    "title": "",
    "seniorityLevel": "",
    "domain": "",
    "employmentType": ""
  },
  "requirements": {
    "mustHaveSkills": [],
    "niceToHaveSkills": [],
    "mustHaveTools": [],
    "mustHaveFrameworks": [],
    "mustHaveDatabases": [],
    "mustHaveCloud": [],
    "mustHaveLanguages": [],
    "mustHaveCertifications": [],
    "minYearsExperience": 0
  },
  "responsibilitySignals": [],
  "screeningQuestions": []
}
```

This extraction can be implemented by:

1. deterministic parsing first if the JD is already structured in the database
2. LLM extraction only for free-text fields such as description and requirements

Best practice:

- split JD requirements into:
  - must-have
  - nice-to-have
  - optional bonus
- add weights per requirement class
- normalize synonyms into canonical skill names

### Phase 4. Normalization layer

Before scoring, normalize CV and JD into comparable tokens.

Examples:

- `ReactJS` -> `React`
- `Golang` -> `Go`
- `Postgres` -> `PostgreSQL`
- `RESTful API` -> `REST API`
- `AWS EC2`, `S3`, `Lambda` -> can map to both detailed skills and umbrella `AWS`

This layer should be mostly deterministic.

Recommended implementation choices:

- internal canonical skill dictionary
- alias table / synonym map
- optional embedding similarity only as secondary support, not primary truth

The result should be:

- `normalizedCvProfile`
- `normalizedJdProfile`

### Phase 5. Deterministic scoring engine

This is the most important change.

The final score should be computed by code, not by LLM output.

Suggested high-level score formula:

```text
Final Score =
  Skill Match Score * 0.50 +
  Experience Match Score * 0.25 +
  Seniority Match Score * 0.10 +
  Domain Match Score * 0.10 +
  Bonus Score * 0.05
```

Example sub-score logic:

1. Skill Match Score
   - must-have skills matched / total must-have skills
   - partial credit for close matches if backed by normalization rules
   - nice-to-have skills contribute smaller bonus

2. Experience Match Score
   - compare candidate years and relevant role history against JD minimum
   - experience directly evidenced in projects or job history gets higher confidence

3. Seniority Match Score
   - compare inferred CV seniority vs JD seniority
   - exact match > near match > mismatch

4. Domain Match Score
   - domain alignment such as FinTech, E-commerce, Healthcare
   - should have lower weight unless the company treats domain as critical

5. Bonus Score
   - certifications
   - leadership
   - architecture experience
   - language requirements

Hard rules can also be added:

- if critical must-have skill coverage < threshold, cap final score
- if minimum language requirement is missing, mark low confidence or hard fail
- if years of experience are far below requirement, apply ceiling

### Phase 6. Explainability output

The API response should explain the score in a machine-readable way.

Example response shape:

```json
{
  "finalScore": 72,
  "scoreLabel": "Good",
  "breakdown": {
    "skillMatchScore": 80,
    "experienceMatchScore": 60,
    "seniorityMatchScore": 80,
    "domainMatchScore": 50,
    "bonusScore": 20
  },
  "evidence": {
    "matchedMustHaveSkills": ["Java", "Spring Boot", "REST API"],
    "missingMustHaveSkills": ["Docker", "Kubernetes"],
    "matchedNiceToHaveSkills": ["AWS"],
    "experienceEvidence": [
      "3 years backend development in Java",
      "built REST services in previous role"
    ],
    "gaps": [
      "No explicit Docker evidence",
      "No production Kubernetes exposure stated"
    ]
  },
  "reasoningSummary": "The candidate matches the core backend stack but misses two operational requirements."
}
```

The summary may be LLM-generated, but only after the deterministic breakdown exists.
The summary should explain the score, not define it.

## Where LLM Should Be Used

Recommended LLM usage:

1. extract structured CV facts from messy CV text
2. extract structured JD requirements from free-text JD
3. normalize ambiguous wording when rules are insufficient
4. generate human-friendly explanation from deterministic scoring output
5. suggest improvement actions for missing requirements

LLM should not be the final authority for:

- final match score
- hard pass/fail decision
- critical compliance requirement checks

## Best Practices For LLM API Usage Without Fine-Tuning

Because this project uses external LLM APIs and does not pretrain or fine-tune a model,
the main objective is to constrain, validate, and audit model behavior.

### 1. Use LLM as extractor, not judge

This is the highest-value practice for recruitment scoring systems.
It is common in production systems to let the model convert raw text into structured facts,
then let application code apply the scoring rules.

### 2. Enforce strict structured output

Prefer:

- JSON schema
- response validation
- retry with error feedback when schema is invalid

Do not rely on free-form text parsing if the output drives business logic.

### 3. Separate extraction from scoring

Do not combine:

- skill extraction
- JD requirement extraction
- score computation
- narrative explanation

into a single prompt.

Each stage should have one clear responsibility.

### 4. Keep prompts narrow and factual

Prompt examples should say:

- extract facts explicitly present or strongly implied
- do not infer unsupported claims
- do not assign a score
- map output to a known schema

Narrow prompts are more stable than broad "analyze everything" prompts.

### 5. Add deterministic validation and post-processing

Examples:

- remove duplicates
- canonicalize aliases
- cap unsupported values
- require evidence for experience claims

### 6. Store intermediate artifacts

Persist at least:

- raw CV text
- structured CV JSON
- structured JD JSON
- normalization output
- scoring breakdown
- final explanation

This makes debugging and product iteration much easier.

### 7. Measure extraction quality

Create an evaluation set of real CV/JD pairs and manually reviewed labels.

Track:

- schema validity rate
- extraction accuracy for must-have skills
- false positive skills
- consistency across repeated runs
- correlation between machine score and recruiter judgment

### 8. Use confidence and fallback rules

When extraction confidence is low:

- mark result as low confidence
- reduce automatic decision weight
- request recruiter review

This is safer than forcing the model to be certain.

### 9. Tune prompts and rules with offline eval, not intuition

For this problem, best practice is:

1. prepare benchmark samples
2. change one prompt/rule at a time
3. compare precision/recall and score stability
4. keep the variant with measurable improvement

### 10. Version prompts and scoring formulas

Every result should be traceable to:

- prompt version
- normalization dictionary version
- scoring formula version

This matters once business users ask why scores changed over time.

## Practical Architecture Options

### Option A. LLM extraction + rule-based scoring

Flow:

1. extract CV structure
2. extract JD structure
3. normalize both
4. compute score by formula
5. generate explanation

Pros:

- explainable
- stable
- easiest to audit
- easiest to tune with product/business teams

Cons:

- requires taxonomy work for skills and requirement types

Recommended as the default approach.

### Option B. LLM extraction + weighted knowledge graph / taxonomy scoring

Flow:

1. extract CV/JD entities
2. map into taxonomy:
   - language
   - framework
   - cloud
   - data
   - tooling
3. score coverage by category and role template

Pros:

- better role-family reuse
- easier to compare backend vs frontend vs data roles differently

Cons:

- more modeling effort

Strong choice if the product will scale to many job families.

### Option C. Embedding-based retrieval + rule-based scoring

Flow:

1. embed CV chunks and JD requirement phrases
2. retrieve likely evidence matches
3. use those matches as support for deterministic scoring

Pros:

- useful for fuzzy wording
- helps surface evidence spans from CV

Cons:

- embeddings alone are not explainable enough for final score
- similarity thresholds need tuning

Best used as a support layer, not the final scoring engine.

### Option D. LLM-as-judge with rubric

Flow:

1. send CV + JD + scoring rubric
2. ask model to assign category scores and final score

Pros:

- fast to prototype

Cons:

- least stable
- hard to audit
- hard to defend to users

Not recommended as the production scoring source of truth.

## Industry-Proven Pattern

For systems that need explainability, auditability, and stakeholder trust, the most
widely accepted pattern is:

1. parse documents into structured facts
2. normalize entities into canonical forms
3. score with deterministic business rules
4. use LLM only for explanation, enrichment, and suggestion

This pattern is used broadly in:

- document intelligence pipelines
- resume parsing products
- underwriting and claims triage support tools
- compliance review assistance systems
- support copilot systems that must show evidence

The reason is simple:

- extraction is probabilistic
- business scoring must be deterministic

## Suggested Scoring Model For SmartCV

Initial version:

```text
50% must-have and nice-to-have skill coverage
20% relevant experience depth
10% seniority alignment
10% domain alignment
10% bonus signals
```

Suggested refinements:

- split skill score:
  - 35% must-have
  - 10% nice-to-have
  - 5% tooling/platform fit
- add score caps:
  - fewer than 40% must-have skills matched => final score capped at 55
  - no core framework match => final score capped at 45
- allow recruiter-configurable role templates later

## Proposed API-Level Flow

### Step 1. Upload CV

Client sends:

- CV file
- `jobId`

### Step 2. Extract CV structure

Service:

- extracts raw text from file
- calls CV extraction prompt
- validates JSON against CV skeleton

### Step 3. Extract JD requirement structure

Service:

- gets JD by `jobId`
- if the job record is already structured, reuse it
- otherwise call JD extraction prompt
- validates JSON against JD skeleton

### Step 4. Normalize

Service:

- maps aliases and synonyms
- prepares comparable features

### Step 5. Score deterministically

Service:

- applies formula and hard rules
- produces breakdown and evidence

### Step 6. Optional explanation generation

Service:

- sends the scoring breakdown to LLM
- asks for a concise explanation and improvement suggestions

### Step 7. Persist artifacts

Store:

- raw CV text
- extracted CV JSON
- extracted JD JSON
- scoring breakdown
- explanation
- prompt/formula versions

## Recommended Prompt Inventory

Instead of one broad scoring prompt, split prompts by responsibility:

1. `extract_cv_structured.md`
   - convert CV text into CV skeleton
2. `extract_jd_requirements.md`
   - convert JD into requirement skeleton
3. `explain_cv_jd_score.md`
   - generate human-readable explanation from deterministic scoring result
4. `improve_cv_against_requirements.md`
   - generate suggestions from known missing requirements

The current `analyze_cv.md` can still be useful during migration, but it should not
remain the source of final score.

## Migration Strategy

### Stage 1

- keep current API contract
- introduce structured CV extraction
- introduce structured JD extraction
- compute a parallel deterministic score
- compare old LLM score vs new formula score offline

### Stage 2

- switch response to deterministic score as source of truth
- keep LLM summary only as explanation

### Stage 3

- add evaluation dataset
- tune weights by recruiter feedback and observed hiring outcomes

---

## Recommended First Implementation Direction

If implementation begins next:

1. create a strict CV skeleton
2. create a strict JD requirement skeleton
3. implement validation and normalization
4. implement a simple deterministic formula
5. generate explanation from the formula output

That gives the fastest path to a system that is:

- understandable
- testable
- measurable
- significantly more trustworthy than direct LLM scoring
