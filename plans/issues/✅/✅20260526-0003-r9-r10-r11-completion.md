## 3. R9 — Payment Service (Mock)

### Goal
Allow a RECRUITER to purchase a quota package; on success, the recruiter's `quotaJobPost` (and optionally `quotaCvViews`) increases atomically with an invoice record.

### Architectural decision
Create a **new `payment_service` microservice on port 8086** (Spring Boot 3.5.x, MongoDB), not a module inside `user-service`. Reasons:
- Keeps `user-service` lean and avoids mixing billing concerns with auth.
- Mirrors the existing service layout (one DB per service).
- Future-friendly: easier to swap mock confirm for VNPay / Stripe later.

### Domain
- `PaymentPackage` (static enum or seeded collection): `BASIC` (5 posts), `PRO` (20 posts), `ENTERPRISE` (-1 = unlimited).
- `Invoice` entity: `id`, `recruiterId`, `packageCode`, `amount`, `quotaJobPost`, `quotaCvViews`, `status` (PENDING / PAID / FAILED), `createdAt`, `paidAt`, `mockReference`.

### Endpoints (gateway path `/payment/**`)
| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| `GET`   | `/api/payments/packages` | RECRUITER | List available packages |
| `POST`  | `/api/payments/orders` | RECRUITER | Create PENDING invoice; returns `orderId` + `mockReference` |
| `POST`  | `/api/payments/confirm/{orderId}` | INTERNAL (gateway secret) | Webhook-style; flips PENDING → PAID and calls user-service to add quota |
| `GET`   | `/api/payments/orders/my` | RECRUITER | Paginated own invoices |
| `GET`   | `/api/payments/orders/{id}` | RECRUITER (owner) / ADMIN | Single invoice |

### Mock confirm flow
After `POST /api/payments/orders`, schedule a 3-second `@Async` task that internally calls `confirm/{orderId}` to simulate a successful payment gateway webhook.

### Atomicity
- Add `POST /internal/recruiters/{id}/quota` to `user-service` (delta-based: `{addJobPosts, addCvViews}`), guarded by `InternalAuthFilter`.
- `payment_service.confirm()` runs inside a MongoDB session/transaction: update Invoice → call user-service. If the HTTP call fails, mark Invoice `FAILED` and emit a `payment.failed` event. (True 2-phase commit is out of scope; this is the documented compensating action.)

## 4. R10 — Recruiter Quota Enforcement in Job Service

### Goal
`job_service.createJob` must verify the recruiter is APPROVED and has remaining `quotaJobPost`, decrement on success, and restore on delete/close.

### Endpoints to add on user-service
| Method | Path | Purpose |
|--------|------|---------|
| `GET`  | `/internal/recruiters/{userId}/profile` | Returns `{status, quotaJobPost, quotaCvViews}` for the recruiter linked to this userId |
| `POST` | `/internal/recruiters/{userId}/consume-job-quota` | Atomic `findAndModify`: if quota>0 OR -1 → decrement, return new quota. Else 409. |
| `POST` | `/internal/recruiters/{userId}/refund-job-quota` | Increment by 1 |

All three require `X-Gateway-Secret` (already enforced by `InternalAuthFilter`).

### Job service changes
| File | Action |
|------|--------|
| `job_service/.../integration/user/UserClient.java` | New — `WebClient` to user-service `/internal/recruiters/**` |
| `job_service/.../integration/user/RecruiterProfile.java` | New record (status, quotaJobPost) |
| `job_service/.../features/job/JobService.java` | In `createJob`: call `getProfile` → if `status != APPROVED` → `RECRUITER_NOT_APPROVED`; call `consumeJobQuota` → if 409 → `QUOTA_EXHAUSTED`. On rollback (exception after consume) → call `refundJobQuota`. |
| `job_service/.../features/job/JobService.java` | In `deleteJob` and `closeJob` (only when first time transitioning to closed/deleted from ACTIVE/DRAFT): call `refundJobQuota`. |
| `job_service/.../enums/ErrorCode.java` | Add `RECRUITER_NOT_APPROVED`, `QUOTA_EXHAUSTED` |
| `job_service/src/main/resources/application.yaml` | Add `app.user-service.base-url` + `app.gateway.internal-secret` |
| `docker-compose.prod.yaml` | Add `USER_SERVICE_URL` env to `job-service` |

### Edge cases
- Unlimited (Enterprise) packages: `quotaJobPost = -1` skips decrement and never refunds.
- Closing a DRAFT (never published) job: still refund (consume happened at create time).
- Idempotency: `refund` is bounded by an upper cap stored on the Recruiter doc (`quotaJobPostMax`) — add this field in user-service to prevent refund loops creating infinite quota; cap = sum of paid PRO/BASIC packages from `payment_service` (or simply ignore refunds beyond `quotaJobPostMax`).

### Acceptance
- New RECRUITER with `status=PENDING` → `POST /api/jobs` returns 403 `RECRUITER_NOT_APPROVED`.
- Admin approves + grants quota 1 → first `POST /api/jobs` succeeds, second returns 409 `QUOTA_EXHAUSTED`.
- DELETE the first job → next `POST /api/jobs` succeeds again.

## 5. R11 — CI/CD Docker Build + EC2 Deploy

### Build & push (GitHub Actions)
For each of the 7 services (`api-gateway`, `user-service`, `job_service`, `application_service`, `notification-service`, `ai_engine_service`, `payment_service`):

1. New workflow `deploy-images.yml` triggered on push to `main`:
   - Detect changed services via `dorny/paths-filter` (avoids rebuilding everything on every commit).
   - For each changed service: `docker/build-push-action@v5` → push to **Docker Hub** (`phamlucchuong/smartcv-<service>:latest` + `:<sha>`).
   - Use `docker/setup-buildx-action` + GHA cache to keep builds <3 min.
2. Secrets required: `DOCKERHUB_USERNAME`, `DOCKERHUB_TOKEN`.
3. Uncomment / remove the dead `# - name: Build Docker Image` blocks in the per-service CI files.

### EC2 deploy
Two options — pick **Option A** for the thesis demo (simpler, no extra infra):

**Option A — SSH + docker compose pull**
- Provision a single EC2 `t3.large` (Ubuntu 22.04, 2 vCPU / 8 GB RAM minimum because Ollama needs ~4 GB just for the 3B model).
- Install Docker + docker compose plugin once via cloud-init.
- Add `deploy-ec2.yml` workflow (runs after `deploy-images.yml` succeeds):
  - Uses `appleboy/ssh-action` with secret `EC2_SSH_KEY`, `EC2_HOST`, `EC2_USER`.
  - Steps: `cd /opt/smartcv && git pull && docker compose -f docker-compose.prod.yaml pull && docker compose -f docker-compose.prod.yaml up -d`.
- Caddy or Nginx in front for SSL via Let's Encrypt (`caddy` is one-liner; pick that).
- Map DNS `api.smartcv.<domain>` → EC2 elastic IP.

**Option B — k3s** (only if you want to mention Kubernetes in the thesis). Adds 4–6 hours; not recommended given timeline.

## 8. Risk Register

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Ollama OOM on EC2 (3B model on `t3.large`) | High | Use `t3.xlarge` (16 GB) or pre-pull `phi3:mini`. Document min specs in `deploy/ec2-bootstrap.sh`. |
| Docker Hub rate-limits anonymous pulls on EC2 | Medium | `docker login` once via systemd unit using stored token. |
| Mongo transaction across `payment_service` + `user-service` is not atomic | Medium | Document compensating action; surface `FAILED` invoices via admin endpoint so they can be retried manually. |
| Quota race condition on concurrent `createJob` | Medium | Use Mongo `findAndModify` with conditional `$inc` (`{$expr:{$gt:["$quotaJobPost",0]}}`); rejection returns 409. |
| CI build times balloon | Low | Cache Maven (`actions/cache`) + Docker buildx GHA cache. |
| EC2 SSH workflow leaks secrets in logs | Low | Use `appleboy/ssh-action`, never echo `.env`. |
