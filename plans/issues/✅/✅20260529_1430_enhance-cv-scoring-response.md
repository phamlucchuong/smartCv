# [feat] Enhance AI CV Scoring: Gap Categorization & Passing Rate Estimate

## Overview

The existing `POST /api/ai/analyze` endpoint returns a flat `missingSkills` list with no distinction
between blocker gaps and optional gaps, and provides no estimate of whether the CV would pass HR
screening. This feature enriches the scoring response with:

1. **Gap categorization**: Split `missingSkills` into `criticalSkills` (blockers that would likely
   cause rejection) and `niceToHaveSkills` (absent but not alone disqualifying).
2. **Passing rate estimate**: Add `passingRateEstimate` (0–100) — AI-estimated probability the CV
   passes the initial HR screening round for this specific job.

Both new fields must propagate through the async scoring pipeline:
`ai_engine_service` → RabbitMQ → `application_service` (stored on the `Application` MongoDB
document) so that recruiter-facing views can display the enriched data.

---

## Reproduction steps

1. Submit `POST /api/ai/analyze` with a CV and a `jobId` whose JD lists both mandatory skills
   (e.g., Docker, Kubernetes) and optional/nice-to-have skills (e.g., Grafana, Prometheus).
2. Observe: the response contains a flat `missingSkills` list — no way to distinguish
   "rejection-level absence" from "bonus skill that would help".
3. Observe: no `passingRateEstimate` field exists; clients cannot surface a
   "chance of passing HR screening" metric to the candidate.

---

## Expected behavior

`POST /api/ai/analyze` response includes the new fields and the flat `missingSkills` is removed:

```json
{
  "matchScore": 68,
  "scoreLabel": "Fair",
  "passingRateEstimate": 42,
  "matchedSkills": ["Java", "Spring Boot", "PostgreSQL"],
  "criticalSkills": ["Docker", "Kubernetes"],
  "niceToHaveSkills": ["Grafana", "Prometheus"],
  "extraSkills": ["Ruby"],
  "summary": "Strong Java background but missing container orchestration skills that are blockers for this DevOps-oriented role."
}
```

The internal callback `PATCH /api/applications/{id}/ai-score` also accepts and persists the enriched
fields so `ApplicationDetailResponse` can expose them to recruiters and candidates.

## Implementation specification

### Phase 1 — ai_engine_service

#### 1. `CvAnalysisResponse.java`

Remove `List<String> missingSkills`. Add three new fields:

```java
public record CvAnalysisResponse(
    int matchScore,
    String scoreLabel,
    int passingRateEstimate,   // NEW: 0–100 estimated HR screening pass probability
    List<String> matchedSkills,
    List<String> criticalSkills,    // NEW: replaces missingSkills — blockers
    List<String> niceToHaveSkills,  // NEW: absent but not alone disqualifying
    List<String> extraSkills,
    String summary
) {}
```

#### 2. `analyze_cv.md` and `skill.md` prompts

The current `analyze_cv.md` has 6 numbered steps:
1. Read the CV and identify skills
2. Compare against job requirements
3. Classify skills as matched / **missing** / extra
4. Calculate `matchScore`
5. Write `summary`
6. Set `scoreLabel`

The current `skill.md` rule #4 defines: *"Missing = in JD requirements but completely absent from CV"*.

**Required changes:**

**`analyze_cv.md`** — Replace step 3 and renumber steps 5–8:

- **Step 3 — Classify skills** (replace): Classify each skill as:
  - `matched`: present in both CV and JD (near-synonym matching applies, see system rules).
  - `criticalSkills`: required by the JD but absent from the CV, **and** whose absence alone would likely
    cause rejection — explicitly labelled "required", listed as a core job-title skill, or repeated
    multiple times in the JD.
  - `niceToHaveSkills`: mentioned in the JD but absent from the CV, and **not** a blocking
    disqualifier alone — e.g., "nice to have", optional certifications, peripheral tools.
  - `extra`: present in the CV but not mentioned anywhere in the JD.
- **Step 5 — Categorize missing skills** (new, insert after step 4):
  Every skill that is absent from the CV must appear in exactly one of `criticalSkills` or
  `niceToHaveSkills` based on the classification in step 3. No skill should remain in a flat
  `missingSkills` list.
- **Step 6 — Estimate passing rate** (new, insert after step 5):
  Compute `passingRateEstimate` (0–100): the probability the CV passes an initial HR screening for
  this job. Use:
  - `matchScore` (primary driver)
  - Number of `criticalSkills` missing: each absent critical skill reduces the estimate by 10–15 points
  - Experience-level alignment: misalignment of ≥2 years penalises by up to 15 points

  Calibration table:

  | passingRateEstimate | Interpretation |
  |---|---|
  | 0–29 | Very unlikely to pass screening |
  | 30–49 | Low chance |
  | 50–69 | Moderate chance |
  | 70–84 | Good chance |
  | 85–100 | Very likely to pass |

- **Step 7 — Write `summary`** (was step 5, renumbered)
- **Step 8 — Set `scoreLabel`** (was step 6, renumbered)

**`skill.md`** — Replace rule #4 "Skill matching logic":

Replace the line:
> `Missing = in JD requirements but completely absent from CV (no near-synonym present).`

With:
> `Missing skills are split into two categories:`
> - `criticalSkills`: absent from CV and would likely cause rejection (explicitly required, core to job title, or repeated in JD).`
> - `niceToHaveSkills`: absent from CV but not a sole disqualifier (optional, peripheral, or "nice to have" wording in JD).`

Updated JSON schema output for `analyze_cv.md` (remove `missingSkills`, add three new fields):

```json
{
  "matchScore": <int 0-100>,
  "scoreLabel": <"Poor"|"Fair"|"Good"|"Excellent">,
  "passingRateEstimate": <int 0-100>,
  "matchedSkills": [<string>, ...],
  "criticalSkills": [<string>, ...],
  "niceToHaveSkills": [<string>, ...],
  "extraSkills": [<string>, ...],
  "summary": <string>
}
```

#### 3. `AiScoreResult.java`

Replace `missingSkills` with `criticalSkills`, `niceToHaveSkills`; add `passingRateEstimate`:

```java
public record AiScoreResult(
    int aiScore,
    int passingRateEstimate,
    List<String> matchedSkills,
    List<String> criticalSkills,
    List<String> niceToHaveSkills,
    String aiStatus
) {}
```

#### 4. `CvScoringConsumer.java`

Update the mapping from `CvAnalysisResponse` to `AiScoreResult` to use the new fields. The
constructor arity changes from 4 to 6 arguments in both the success path and the catch block:

```java
// Success path (was 4-arg, now 6-arg)
applicationClient.updateAiScore(message.applicationId(), new AiScoreResult(
    result.matchScore(),
    result.passingRateEstimate(),
    result.matchedSkills(),
    result.criticalSkills(),
    result.niceToHaveSkills(),
    "SCORED"            // keep "SCORED" — matches AiScoringStatus.SCORED enum value
));

// Failure path (was 4-arg, now 6-arg)
applicationClient.updateAiScore(message.applicationId(), new AiScoreResult(
    0, 0, null, null, null, "FAILED"
));
```

### Phase 2 — application_service

#### 5. `AiScoreUpdateRequest.java`

Replace `missingSkills` with `criticalSkills`, `niceToHaveSkills`; add `passingRateEstimate`:

```java
@Data @NoArgsConstructor @AllArgsConstructor
public class AiScoreUpdateRequest {
    int aiScore;
    int passingRateEstimate;
    List<String> matchedSkills;
    List<String> criticalSkills;
    List<String> niceToHaveSkills;
    AiScoringStatus aiStatus;
}
```

#### 6. `Application.java` entity

Remove `missing_skills` field. Add:

```java
@Field("passing_rate_estimate")
Integer passingRateEstimate;

@Field("critical_skills")
List<String> criticalSkills;

@Field("nice_to_have_skills")
List<String> niceToHaveSkills;
```

#### 7. `ApplicationService.java`

In the method that handles `PATCH /api/applications/{id}/ai-score`, **replace** the existing
`setMissingSkills` call with the three new setters. Both the setter on `Application` and the
getter on `AiScoreUpdateRequest` will no longer exist after steps 5–6, so this line must be
removed to avoid a compile error:

```java
// Remove this line:
application.setMissingSkills(request.getMissingSkills());

// Replace with:
application.setPassingRateEstimate(request.getPassingRateEstimate());
application.setCriticalSkills(request.getCriticalSkills());
application.setNiceToHaveSkills(request.getNiceToHaveSkills());
```

#### 8. Response DTOs

Add `passingRateEstimate`, `criticalSkills`, `niceToHaveSkills` to both
`ApplicationDetailResponse.java` and `ApplicationResponse.java` so the enriched data
is returned to API consumers (recruiter and candidate views).

#### 9. `ApplicationMapper.java`

No new `@Mapping` annotations are needed — MapStruct auto-maps fields by matching property names,
so the three new fields will be generated automatically once they exist on both the `Application`
entity and the response DTOs. Run `./mvnw clean compile` after this change so that MapStruct
regenerates `ApplicationMapperImpl` without the now-removed `missingSkills` property; an
incremental build may retain the old generated class.

## Notes

- The `/api/ai/improve` endpoint (`CvImprovementResponse`) is **out of scope** for this issue;
  its existing strengths/weaknesses/tips output already covers "how to improve" guidance.
- `passingRateEstimate` calibration rules must be documented inside the prompt, not in code, so
  future prompt iterations can adjust the curve without a code deploy.
- The cross-service `aiStatus` wire format is intentionally asymmetric: `AiScoreResult.aiStatus`
  is a `String` (e.g., `"SCORED"`) so the ai_engine_service has no compile-time dependency on
  the application_service enum. `AiScoreUpdateRequest.aiStatus` deserializes that string into the
  `AiScoringStatus` enum. This contract is unchanged by this feature.
- MongoDB is schema-less; existing `Application` documents with `missing_skills` are untouched.
  The new fields will be absent (null) on old documents — response DTOs must handle null gracefully
  (return empty lists, not NPE).
- api-gateway routing (`/ai/**` → ai_engine_service port 8085) already exists; no gateway changes
  are required.
