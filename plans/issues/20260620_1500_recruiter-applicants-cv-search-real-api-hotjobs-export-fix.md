# Recruiter Applicants & CV Search — Real API + Fix useGetHotJobs Export

**Date:** 2026-06-20
**Type:** Feature + Bug Fix

---

## Overview

Two separate issues bundled together:

1. **Bug** — `packages/api` emits `SyntaxError: does not provide an export named 'useGetHotJobs'` on the web-candidate dev server because `index.ts` had two `export *` sources that both re-exported that name, making it ambiguous under the ESM spec.

2. **Feature** — The web-recruiter applicants page (`/employer/applicants`) and CV search page (`/employer/cv-search`) are fully wired to static mock data. They need to be connected to real backend APIs.

---

## Part 1 — Bug: `useGetHotJobs` Not Exported (SyntaxError)

### Root Cause

`packages/api/src/index.ts` (commit `8ff4fad`) contained two `export *` statements that both re-exported `useGetHotJobs`:

```ts
// line 49
export * from './generated/job/home-controller/home-controller';   // ← exports useGetHotJobs

// line 64
export * from './home-hooks';                                       // ← also exports useGetHotJobs
```

Per the ESM specification: when the same name is re-exported from two or more `export *` sources, the name is treated as **ambiguous** and is excluded from the module namespace. Vite's dev server exposes this as a `SyntaxError` at the consumer import site:

```
Uncaught SyntaxError: The requested module '...packages/api/src/index.ts'
does not provide an export named 'useGetHotJobs' (at index.tsx:25:3)
```

`home-hooks.ts` was a hand-written hook file created before `home-controller.ts` was generated by Orval. Once the generated file covered the same hook, `home-hooks.ts` became a duplicate.

### Fix

**Step 1 — Already applied** (commit `d0c6012`): remove `export * from './home-hooks'` from `index.ts`.
The ambiguity is gone; `useGetHotJobs` is now provided unambiguously from `home-controller.ts`.

**Step 2 — Cleanup needed**: `home-hooks.ts` is now dead code. Delete the file to avoid future confusion.

**Step 3 — Dev server restart**: Vite's module graph caches the old resolution. After the code change, run:
```bash
pnpm -F web-candidate dev   # kill and restart
```

### Files to Change

| File | Change |
|---|---|
| `packages/api/src/home-hooks.ts` | Delete (entirely superseded by generated home-controller) |
| Dev server | Restart to clear Vite module cache |

### Backend APIs Available

#### Application Service

| Endpoint | Hook | Auth | Notes |
|---|---|---|---|
| `GET /application/api/applications/job/{jobId}?page&size` | `useGetByJobId` | `RECRUITER` | Paginated applications for one job |
| `GET /application/api/applications/{id}` | `useGetById` | Authenticated | Single application detail |
| `PATCH /application/api/applications/{id}/status` | `useUpdateStatus` | `RECRUITER` | Update status + notes |

**`ApplicationDetailResponse` shape:**
```ts
{
  id: string
  candidateId: string       // Candidate profile _id (user-service)
  jobId: string
  recruiterId: string
  status: ApplicationStatus // PENDING | REVIEWING | ACCEPTED | REJECTED | WITHDRAWN
  coverLetter?: string
  cvUrl: string
  rejectionReason?: string
  recruiterNotes?: string
  aiScore?: number
  matchedSkills: string[]
  missingSkills: string[]
  aiStatus: AiScoringStatus
  appliedAt: string
  updatedAt: string
}
```

> **Important:** `ApplicationDetailResponse` contains only `candidateId`. There is no `candidateName`, `candidateTitle`, or `candidateAvatarUrl`. Candidate info must be fetched separately from user-service (see below).

**`ApplicationStatusUpdateRequest` shape:**
```ts
{ status: ApplicationStatus; rejectionReason?: string; recruiterNotes?: string }
```

#### User Service — Candidate Controller

| Endpoint | Hook | Auth | Notes |
|---|---|---|---|
| `GET /user/api/candidates/{id}` | `useGetById2` | Authenticated | Single candidate profile by candidate `_id` |
| `GET /user/api/candidates?page&size` | `useGetAllCandidates` (alias for `useGetAll2`) | `RECRUITER` or `ADMIN` | Paginated candidate list |

**`CandidateResponse` shape (relevant fields):**
```ts
{
  id: string            // candidate profile _id (matches ApplicationDetailResponse.candidateId)
  userId: string
  fullName: string
  email: string
  phone?: string
  title?: string
  avatarUrl?: string
  skills: string[]
  yearsOfExperience?: number
  address?: string
  bio?: string
}
```

#### Job Service

| Endpoint | Hook | Auth | Notes |
|---|---|---|---|
| `GET /job/api/jobs/mine?page&size` | `useGetMyJobs` | `RECRUITER` | Recruiter's own job list for filter dropdown |

### 2a. Applicants Page (`employer.applicants.tsx`)

**Goal:** Replace mock data with real paginated applications grouped by job.

#### Implementation Plan

1. **Job filter dropdown:** Call `useGetMyJobs()` (enabled when recruiter is authenticated) to populate the "chọn tin tuyển" filter. Store `selectedJobId` in state (default: first job or `undefined` = "Tất cả").

2. **Application list:** Call `useGetByJobId(selectedJobId, { page, size })` when a job is selected.
   - If no job selected, show a prompt to select a job first (there is no "all applications across all jobs" endpoint).

3. **Candidate enrichment:** `ApplicationDetailResponse` lacks candidate name/avatar. For each application returned, call `useGetById2(application.candidateId)` to fetch the candidate profile. Since this is an N+1 pattern, batch the lookups:
   - Use a single `useGetAllCandidates({ page: 1, size: 100 })` query and build a local lookup map `Map<candidateId, CandidateResponse>` — acceptable for moderate application volumes.
   - OR: call `useGetById2` per application with `staleTime: Infinity` so React Query deduplicates repeated lookups.
   - Recommended: **per-application `useGetById2`** with `staleTime: 60_000` — simpler, no over-fetching of irrelevant candidates.

4. **Status mapping:** The real `ApplicationStatus` enum is `PENDING | REVIEWING | ACCEPTED | REJECTED | WITHDRAWN`. The kanban `STATUS_COLUMNS` array must be updated to these real values (replacing the mock's `Qualified`, `Interview Scheduled`, etc.).

5. **Kanban drag-and-drop:** `handleDrop` currently mutates local state. Replace with `useUpdateStatus.mutate({ id, data: { status: newStatus } })` and reflect the result in local state on success.

6. **Loading and empty states:** Show skeleton cards while `useGetByJobId` is loading. Show "no applications" message when the list is empty.

#### Hook call patterns

```ts
// Job filter — first param is GetMyJobsParams (pass undefined for defaults), second is options
const { data: myJobsData } = useGetMyJobs(undefined, { query: { enabled: isRecruiter } })
const myJobs = myJobsData?.data?.items ?? []

// Applications for selected job
const { data: applicationsData, isLoading } = useGetByJobId(
  selectedJobId ?? '',
  { page, size: 50 },
  { query: { enabled: !!selectedJobId } }
)
const applications = applicationsData?.data?.items ?? []

// Status update
const updateStatusMutation = useUpdateStatus()
updateStatusMutation.mutate({ id: applicationId, data: { status: 'REVIEWING' } })
```

### 2b. Applicant Detail Page (`employer.applicants.$id.tsx`)

**Goal:** Replace mock CANDIDATES lookup with real application + candidate data.

The route param `$id` currently matches a `Candidate.id` in mock data. With real API, `$id` should be the **application `id`** (not the candidate id).

#### Implementation Plan

1. Call `useGetById(id)` (from application-controller — `GET /api/applications/{id}`) to load the `ApplicationDetailResponse`.
2. From that, call `useGetById2(application.candidateId)` to load the `CandidateResponse`.
3. Render candidate info (`fullName`, `title`, `skills`, `email`, `phone`, `avatarUrl`) from `CandidateResponse`.
4. Render application info (`aiScore`, `matchedSkills`, `missingSkills`, `status`, `cvUrl`, `appliedAt`) from `ApplicationDetailResponse`.
5. "Chuyển stage" button: open a status picker, call `useUpdateStatus.mutate`.
6. "Từ chối" button: call `useUpdateStatus.mutate` with `{ status: 'REJECTED', rejectionReason }`.

> **Navigation change:** `employer.applicants.tsx` currently links to the detail page via `params={{ id: c.id }}` at **lines 186, 261, and 295**, where `c.id` is the mock `Candidate.id`. Replace all three with `params={{ id: application.id }}` (the real application document id returned by `useGetByJobId`). The route param `$id` on the detail page must then be treated as an application id throughout.

### 2c. CV Search Page (`employer.cv-search.tsx`)

**Goal:** Replace mock data with real candidate list from user-service.

#### Implementation Plan

1. Call `useGetAllCandidates({ page: 1, size: 100 })` to load candidates (recruiter role required).

2. **Client-side filtering:** The backend `GET /api/candidates` only supports `page/size` — no keyword/skill/experience/location params. Filter client-side using the `CandidateResponse` fields:
   - keyword → `fullName`, `title`, `bio`
   - skill → `skills[]`
   - experience → `yearsOfExperience`
   - location → `address`

3. **Pagination:** `ITEMS_PER_PAGE = 5` — after client-side filter, slice and page the result locally.

4. **CV Unlock:** Keep the local `unlockedIds` state pattern for now (no service plan API). When a candidate is "unlocked", show their `email` and `phone` from `CandidateResponse`.

5. **CV download link — not implementable with current APIs.** `CandidateResponse.cvUrl` exists in the DTO but `CandidateMapper` ignores it (`@Mapping(target = "cvUrl", ignore = true)`), so it is always `null` in both list and single-fetch responses. The `GET /api/candidates/cvs` endpoint (`useListCvs`) is scoped to the authenticated candidate themselves (`@AuthenticationPrincipal`) — recruiters cannot call it. **Conclusion:** the CV download/view action in the unlock modal must be hidden or disabled until a recruiter-accessible CV endpoint is added to user-service (out of scope for this issue).

#### Hook call pattern

```ts
const { data: candidatesData, isLoading } = useGetAllCandidates(
  { page: 1, size: 100 },
  { query: { enabled: isRecruiter } }
)
const allCandidates = candidatesData?.data?.items ?? []
```

## Notes

- The `ApplicationStatus` enum on the backend is `PENDING | REVIEWING | ACCEPTED | REJECTED | WITHDRAWN`. The kanban column labels and drag-and-drop logic must use these exact values, not the mock's human-readable strings.
- `useGetByJobId` requires a `jobId` to be enabled. If the recruiter selects "all jobs", iterate over each job and aggregate, or display a per-job tabs UI instead of a flat list.
- For the CV search page, a recruiter cannot see a candidate's CV URL directly from the list endpoint — it requires fetching the individual candidate profile. This is acceptable since CV access is gated behind the "unlock" action.
- `home-hooks.ts` can be deleted safely; it was a temporary hand-written hook before the Orval generation covered the home controller.
