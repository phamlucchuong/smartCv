# Assessment Manager — Wire candidate take-flow and recruiter assign dialog to real APIs

## Overview

The backend Assessment domain (`application_service`, 9 endpoints) and `@smart-cv/api` hooks (generated by Orval) are fully implemented and production-ready. Two frontend gaps remain that prevent the feature from working end-to-end:

1. **web-candidate** — `TakeAssessment` UI (`_account.assessments.tsx`) uses a hardcoded `QUESTIONS` array and never calls any real API (`useSaveAnswers`, `useSubmitAttempt`, `useGetResult`). The submitted result always displays "85/100" regardless of actual answers. Assessment cards also show a raw UUID (`assessmentId`) instead of the human-readable title.

2. **web-recruiter** — the "Gán cho ứng viên" (Assign to candidate) dialog and the job selector in the Create/Edit form both read from static `mock-data.ts` (`CANDIDATES`, `JOBS`) instead of real APIs.

**Important backend invariant**: `assignToCandidate` in the backend immediately creates an `IN_PROGRESS` attempt when the recruiter assigns an assessment. There is no "assigned but not started" state — the candidate's `AttemptStateResponse` will always have a real `attemptId` and `status = IN_PROGRESS` from the moment of assignment. Therefore `useStartAttempt` is not needed for the recruiter-assigned flow.

## Reproduction steps

1. Recruiter opens `employer.assessments.tsx` → clicks "Gán" on any assessment → dialog shows hard-coded candidate names from `mock-data.CANDIDATES`, not real applicants.
2. Recruiter opens Create/Edit assessment form → job selector shows hard-coded jobs from `mock-data.JOBS`.
3. Candidate logs in → navigates to `/account/assessments` → each card displays a raw UUID as the assessment name.
4. Candidate clicks "Tiếp tục" (all assigned assessments show `IN_PROGRESS`) → sees three hard-coded Java interview questions instead of real questions.
5. Candidate answers and submits → no API call is made; result is always "85/100".

## Expected behavior

- Candidate cards show the real assessment title (fetched from `useGetAssessment(assessmentId)`).
- Clicking "Tiếp tục" opens the take-flow, loads real questions via `useGetAssessment(assessmentId)`, restores any saved answers from `useGetAttemptState(attemptId)`.
- Answering questions saves progress via `useSaveAnswers`; submitting calls `useSubmitAttempt` → result shown from `useGetResult`.
- Question UI matches recruiter preview: question-map panel (jump-to-question), MCQ radio options, TEXT textarea, countdown timer, progress bar.
- Recruiter Assign dialog shows real applicants filtered by the assessment's `jobId` (via `useGetRecruiterApplications`).
- Recruiter Create/Edit form job selector shows recruiter's real jobs from a manual `useGetRecruiterJobs` hook.

## Current behavior

- Candidate `TakeAssessment` component: reads hardcoded `QUESTIONS` constant defined in the same file; no real API calls for save/submit.
- Candidate cards: display `a.assessmentId` (raw UUID) as the assessment name.
- Recruiter Assign dialog: imports and renders `CANDIDATES` from `@/lib/mock-data`.
- Recruiter job selector: imports `JOBS` from `@/lib/mock-data`.

## Implementation plan

### Task 1 — web-candidate: Fix assessment card title display

`AttemptStateResponse` carries `assessmentId` but not a title.

**Option A — pure frontend (recommended, no backend change):**
Extract each card into an `AssessmentCard` sub-component that calls `useGetAssessment(a.assessmentId)` and shows `assessment.data.title`. React Query caches the result — multiple cards with the same `assessmentId` share one network call.

```tsx
function AssessmentCard({ attempt }: { attempt: AttemptStateResponse }) {
  const { data } = useGetAssessment(attempt.assessmentId!, {
    query: { enabled: !!attempt.assessmentId }
  })
  const title = data?.data?.title ?? attempt.assessmentId
  // render card using `title`
}
```

Note: `useGetAssessment` is restricted to `CANDIDATE` role on the backend — this is fine for the candidate page.

**Option B — cleaner long-term (requires backend change):** Add `assessmentTitle: String` to `AttemptStateResponse` in `application_service` → populate in `AssessmentService.getMyAssessments` → run `pnpm generate:api`.

**Also update the search filter** (line 130 in `_account.assessments.tsx`): after titles are available, the filter `(item.assessmentId ?? '').toLowerCase().includes(q)` should be updated to search against the loaded title instead of the UUID.

### Task 2 — web-candidate: Rewrite TakeAssessment component

**Props interface change (required):**
```tsx
// Old:
function TakeAssessment({ onClose }: { onClose: () => void })

// New:
function TakeAssessment({ assessmentId, attemptId, onClose }: {
  assessmentId: string
  attemptId: string
  onClose: () => void
})
```

Update the caller in `AssessmentsPage` (line 69 / `setTaking`) to store `{ assessmentId, attemptId }` in state instead of just a string.

**Fetch real questions and saved answers on mount:**
```tsx
const { data: assessmentData } = useGetAssessment(assessmentId)
const { data: attemptData } = useGetAttemptState(attemptId)
const questions = assessmentData?.data?.questions ?? []
const savedAnswers = attemptData?.data?.answers ?? []
```

The backend's `AssessmentService.getAssessment` strips `correctOptionIndex` before returning — do NOT implement client-side grading.

**Backend invariant**: Since `assignToCandidate` creates an `IN_PROGRESS` attempt immediately, `useStartAttempt` is NOT needed. The `attemptId` already exists in `AttemptStateResponse`. The distinction between "Bắt đầu" (first open) and "Tiếp tục" (resume) is:
- Label "Bắt đầu" (Start): `answers.length === 0` — no saved answers yet
- Label "Tiếp tục" (Continue): `answers.length > 0` — restore saved answers

**Question UI** (mirror recruiter preview layout `TakeAssessmentPreview` in `employer.assessments.tsx`):
- Left panel: question-map grid (numbered buttons, colored by answered/active/unanswered state)
- Main panel: question text + MCQ radio list OR TEXT `<textarea>`, progress bar, countdown timer from `assessmentData.timeLimitMinutes`
- Navigation: "Câu trước" / "Câu tiếp theo" + direct jump from question map

**Save answers** (call `.mutate()` not the hook directly):
```tsx
const saveAnswersMutation = useSaveAnswers()
// On answer change or question navigation:
saveAnswersMutation.mutate({
  attemptId,
  data: { answers: currentAnswers }
})
```
`questionId` for each answer must be `question.id` from the loaded `AssessmentResponse.questions` array, not the array index.

**Submit** — `useSubmitAttempt` is a mutation; `useGetResult` is a query (declare at top level, enable conditionally):
```tsx
const submitMutation = useSubmitAttempt()
const { data: resultData, refetch: fetchResult } = useGetResult(attemptId, {
  query: { enabled: false }
})

// On submit button click or timer expiry:
const handleSubmit = () => {
  if (isSubmitting) return  // guard against double-submit (timer race)
  setIsSubmitting(true)
  submitMutation.mutate({ attemptId }, {
    onSuccess: () => { setShowResult(true); fetchResult() }
  })
}
```

**Timer expiry**: when countdown hits 0, call `handleSubmit()`. The `isSubmitting` guard prevents a race if the user clicks "Nộp bài" at the exact moment the timer fires.

**Result display** — handle all three states:
- `result === 'PASS'` → green success, show score
- `result === 'FAIL'` → red, show score  
- `result === 'PENDING'` → amber, message: "Bài thi có câu tự luận sẽ được nhà tuyển dụng chấm điểm thủ công. Kết quả sẽ được cập nhật sau." (add i18n key: `assessment_result_pending_desc`)

### Task 3 — web-recruiter: Fix Assign dialog to use real applicants

In `employer.assessments.tsx`, replace the `CANDIDATES` mock:

1. When the Assign dialog opens (`isAssignOpen === true`), enable `useGetRecruiterApplications({ size: 1000 })`. Passing a large `size` avoids silent page truncation — without it the default page size may omit applicants from the dropdown.
2. Filter the response array by `item.jobId === assessmentToAssign?.jobId` (skip filter if `assessmentToAssign?.jobId` is blank — show all applicants).
3. **`ApplicationDetailResponse` has no candidate name field** (confirmed from generated model: `id, candidateId, jobId, recruiterId, jobTitle, companyName, status, ...`). Display a composite label: `` `${item.candidateId?.slice(0, 8)}… — ${item.jobTitle}` `` as a practical workaround. A cleaner fix (include candidate display name in the DTO) is out of scope for this issue.

Note: `useGetRecruiterApplications` currently has no `jobId` query param — client-side filtering is used. Pagination is handled by the `size: 1000` workaround above.

## Notes

- Gateway: all assessment endpoints route via `/application/**` → `APP_SERVICE_URI:http://localhost:8083`. The gateway's `application.yaml` already covers this — no gateway changes needed.
- Backend security: `AssessmentService.getAssessment` strips `correctOptionIndex` before returning to candidates. Do not implement client-side auto-grading in the candidate UI — scores are computed server-side at submit time.
- PENDING result: assessments with at least one TEXT question always return `result = PENDING` after submit. Display a clear manual-grading message with i18n key `assessment_result_pending_desc`.
- Pass threshold: MCQ-only assessments use 70% cutoff (`score >= 70 → PASS`), defined in backend `AssessmentService.submitAttempt`. Frontend must not replicate this logic.
- Backend `ATTEMPT_ALREADY_IN_PROGRESS` error: since `assignToCandidate` always creates an `IN_PROGRESS` attempt, `useStartAttempt` should not be called from the candidate UI for recruiter-assigned assessments — it would fail with this error. The `attemptId` from `AttemptStateResponse` is used directly.
- `useGetAssessment` requires `CANDIDATE` role. It is not callable by recruiters (they receive 403). The recruiter preview (`TakeAssessmentPreview`) already correctly works from local `assessment` prop data, not the API.
- All Orval-generated mutation hooks must be called via `.mutate({ ... })` or `.mutateAsync({ ... })` on the mutation object returned by the hook, not called with params directly.
